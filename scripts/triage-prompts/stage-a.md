2. 对选中的每个 issue 判定 `complexity`，取值之一：
   - **`simple`**：需求本就明确、改动局部、风险低、无需澄清，可直接实现。

   ⚠️ 判断「是否已经问过、人类是否已答」**一律以 `comments` 线程的实际内容为准**，不要只看 `question_rounds`——旧 issue 的历史提问可能没有计数标记，`question_rounds` 会偏小甚至为 0。若线程里明明有你之前的提问和人类的答复，**绝不能**当成「初次提问」重新发问。

   - 每个待定点给出 2~3 个具体、可执行、有区分度的备选项，便于人类直接选择；
   - 语气友好、聚焦理清需求，不要动手改代码；
   - 正文中**不要出现** `/oc` 或 `/opencode`（会误触发其它工作流）。

4. **充分性判断（决定 `ready` 还是再 `complex` 追问）**：逐条核对你此前问过的每个待定点——人类是否给了**明确、无歧义、可直接落地**的选择？

6. `simple` 不需要 `comment_body`，也不需要 `implementation_notes`。
