# Reflow — 单板 BOM 状态管理工具 · 需求文档

> 工具名「Reflow」一语双关：既是硬件的回流焊（reflow soldering），又描述本工具的核心机制——编辑上游节点后，改动会向后「重新流动」传播。
>
> 本文档用于交给 Claude Code 进行开发。先读「核心概念」与「版本模型」，这两节是整个工具的地基。

## 1. 背景与目标

硬件调试过程中，需要记录每块单板在每次测试时的 BOM 状态，并像 git 一样对 BOM 的演进做版本管理。每个状态都有一个稳定、可分享的链接，外部笔记软件里的测试记录会引用这些链接，从而保证「每次测试用的是哪个 BOM 状态」始终可追溯。

因为是硬件，单板的演进是**线性的，没有分支**。

## 2. 核心概念与数据层级

单板通过四级定位：`单板名称` → `PCB版本` → `BOM版本` → `单板ID`。

- **初始 BOM 绑定在 `BOM版本` 这一层**：同一个 `BOM版本` 下的所有单板共享同一份初始 BOM。
- **调试历史（状态图）绑定在 `单板ID` 这一层**：每块实物板从共同的初始 BOM 出发，各自独立演进，互不影响。

BOM 条目只有两个有效字段：

- `Reference`（位号，如 `R1`、`C86_PD1`），唯一标识一个元件位置。
- `Part`（器件型号，如 `1kR`、`100nF`、`HE364B-G`）。

**不贴 (DNP) 的表示**：某个位号「不贴」就是它**不出现在该状态的 BOM 中**。没有单独的贴装标记字段——「无此位号」即等于「不贴」。

## 3. 版本模型（关键，差量存储）

- 每块单板从**初始 BOM（根节点）** 开始。
- 每个**节点 = 一次 commit = 一个 BOM 状态**。
- 节点**只存储相对父节点的修改集（changeset）**，而不是整份 BOM 快照。
- 任意节点的完整 BOM = 初始 BOM + 沿链按顺序叠加每个节点的修改集。
- 修改集中针对单个位号的操作有三种：
  - **新增**：增加一个原本不存在的位号。
  - **修改**：改变某位号的 `Part`。
  - **删除**：去掉某位号（即「不贴」）。
- **工作区**：最新节点之后挂一个工作区，调试中的改动先在工作区里累积，到阶段了再 commit 生成新节点。
- **commit 元数据**：说明文字 + 时间戳。
- 结构线性，无分支、无合并。

## 4. 历史编辑与传播（关键，含冲突处理）

### 4.1 编辑的语义约定

编辑历史节点的用途是**纠正记录错误**（"这个位号当时本来就是这个值，我之前记错/漏记了"），而**不是表示一次新的改动**。真正的新改动应该在链的末端 commit 成新节点。工具可以不强制这条约定，但 UI 文案应体现这是「修正」。

采用「就地覆盖」方式（不为每次编辑生成新版本号），可追溯性由第 5 节的审计日志兜底。

### 4.2 传播规则

由于采用差量存储，传播在很大程度上是**自动**的：

- 编辑某中间节点的某个位号后，所有**继承**该位号值的后续节点（即它们自己的修改集里没碰过这个位号），重新计算完整 BOM 时会自动跟着变——无需额外操作。
- **冲突**仅发生在：某个后续节点的修改集里**显式操作过同一个位号**。这时该节点（及其之后）仍解析为它自己的值，需要停下来让用户逐位号确认。

### 4.3 冲突确认

对每一个冲突位号，让用户二选一：

- **保留下游节点自己的值**（那是一次真实的后续改动，与本次修正无关）；
- **采用本次修正的值并向后传播**（移除下游节点对该位号的显式操作，让修正流过去）。

新增、修改、删除三种操作都同样参与传播与冲突判定。

### 4.4 一个具体例子（实现时请对照）

初始 BOM：`R1 = 10k`。链为 `初始 → S1 → S2 → S3`。

- 当前 `S2` 的修改集里有一条 `R1 改为 47k`；`S1`、`S3` 没碰过 `R1`。
- 现在用户打开 `S1`，把 `R1` 修正为 `22k`（即在 `S1` 的修改集里写入 `R1 = 22k`）。
- 结果：
  - `S1` 解析为 `R1 = 22k`。
  - `S3` 继承自最近的显式值，`S2` 仍是 `47k` → 与本次修正冲突，弹出确认。
  - 若用户选「保留下游 47k」：`S2`、`S3` 仍为 `47k`，仅 `S1` 变为 `22k`。
  - 若用户选「采用修正值」：移除 `S2` 对 `R1` 的显式操作，`S1`/`S2`/`S3` 全部解析为 `22k`。

## 5. 审计日志（append-only）

- **每次编辑追加一行，永不覆盖**（一个节点被改 10 次就有 10 条记录）。
- 每行字段：节点、位号、`Part` 旧值 → 新值（含新增/删除的标记）、时间戳、**来源**（直接编辑 / 上游传播而来）、可选的原因备注。
- 「来源」用于解释「为什么这个节点在某次测试之后变了」（例如：由上游某节点的编辑传播而来）。
- **暂不做「按时间回看」界面**，日志先仅用于审计与查询；但数据结构应保证将来能据此回放任意时间点的状态。

## 6. CSV 导入

用于建立某个 `BOM版本` 的初始 BOM（即根节点）。

附件示例的列为：`Item, Quantity, Reference, Part, PCB Footprint, Assembly Type, Linter Info`。

导入规则：

- **只取 `Reference` 和 `Part` 两列**，其余全部忽略。
- `Reference` 列常把多个位号合并在一格、用逗号分隔并加引号（如 `"R30,R25,R31,R26"`），共用同一个 `Part`。导入时必须**拆成一位号一条**，每条形如 `(Reference, Part)`。例：`"R67,R24" | 1kR` → `R67=1kR`、`R24=1kR`。
- 健壮性处理：UTF-8 BOM 头、CRLF 行尾、含逗号的带引号字段、位号首尾空格。
- 位号可能带下划线/后缀（如 `C86_PD1`），按字符串原样处理，不做解析。

## 7. 页面与交互

### 7.1 层级导航与新建

- 可浏览/选择 `单板名称 → PCB版本 → BOM版本 → 单板ID`。
- 新建 `BOM版本`：上传 CSV 建立其初始 BOM。
- 新建 `单板ID`：在已有 `BOM版本` 下创建，**自动继承该版本的初始 BOM 作为根节点**，之后独立演进。

### 7.2 状态图页面（git graph 风格）

- 针对**单个 `单板ID`**，用线和节点线性展示其状态演进。
- 每个节点显示说明文字与时间戳。
- 点击节点进入节点详情页。

### 7.3 节点详情 / 编辑页

- 显示该状态的**完整 BOM**（`Reference` + `Part` 列表）。
- **diff 高亮**：相对上一个节点，标出本节点改动的位号——新增 / 修改 / 删除分别用不同样式标识。
- 可编辑：修改某位号的 `Part`、新增位号、删除位号（不贴）。
- 任何编辑触发第 4 节的传播 + 冲突确认流程。
- 该页面拥有**稳定 URL**（见第 8 节）。

### 7.4 工作区与 commit

- 在最新节点之上累积改动（增/改/删位号）。
- commit 时填写说明文字，生成新节点。

### 7.5 冲突确认界面

- 列出所有冲突位号，逐个让用户选择「保留下游值」或「采用修正值」。

## 8. 稳定链接

- 每个节点拥有稳定、可分享的 URL，例如 `/board/{单板ID}/node/{nodeId}`。
- 在外部笔记软件中粘贴该链接，点开即可看到对应状态的完整 BOM。
- URL 在节点生命周期内保持不变（编辑节点内容不改变其 URL）。

## 9. 技术约束

- **单人使用**，暂不考虑多用户与并发。
- 数据库使用 **SQLite**。
- 因为需要稳定可分享 URL，实现为 **Web 应用**。

## 10. 建议数据模型（供实现参考，可调整）

- `boards`：单板层级信息（单板名称、PCB版本、BOM版本、单板ID 的组织方式，可按需拆表或用一张表）。
- `bom_versions` / `initial_bom`：每个 BOM版本 的初始 BOM 条目（Reference, Part）。
- `nodes`：节点/commit（id、所属单板ID、父节点、说明文字、时间戳）。
- `node_changes`：节点的修改集条目（节点id、Reference、操作类型 add/modify/remove、Part 值）。
- `edit_log`：审计日志（节点id、Reference、旧值、新值、操作、来源 direct/propagated、时间戳、备注）。

完整 BOM 通过「初始 BOM + 沿父链折叠所有 `node_changes`」在读取时计算得出。

## 11. MVP 边界

**本期要做**：CSV 导入建初始 BOM、单板层级与新建、单板状态图、节点查看 + diff 高亮、工作区与 commit、历史节点编辑 + 传播 + 冲突确认、append-only 审计日志、稳定链接。

**暂不做（预留扩展）**：

- 飞线等「位号以外」的修改类型（后续可能加，数据模型留好扩展空间）。
- 「按时间回看 / 状态回放」界面。
- 多用户、并发、权限。
- 工具内的测试记录功能（测试记录在外部笔记软件，本工具只提供链接）。
