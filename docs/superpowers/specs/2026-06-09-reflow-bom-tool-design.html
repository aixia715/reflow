Reflow — 单板 BOM 状态管理工具

像 git 一样对硬件单板的 BOM 演进做线性版本管理。每个状态有稳定可分享的 URL，供外部笔记引用，保证「每次测试用的是哪个 BOM 状态」始终可追溯。工具名一语双关：硬件回流焊，也描述核心机制——编辑上游节点后，改动向后「重新流动」传播。

1 · 目标与范围

硬件调试中，记录每块单板每次测试时的 BOM 状态，并对其演进做线性版本管理（无分支、无合并）。单人使用，不考虑多用户与并发。数据库用 SQLite；因需稳定 URL，实现为 Web 应用。

3 · 核心概念 & 版本模型

3.1 数据层级

单板通过四级定位：单板名称 → PCB版本 → BOM版本 → 单板ID。

  • 初始 BOM 绑定在 BOM版本 层：同一 BOM版本 下所有单板共享同一份初始 BOM。
  • 调试历史（状态图）绑定在 单板ID 层：每块实物板从共同初始 BOM 出发，各自独立演进。
  • 单板名称 与 PCB版本 不是独立实体表，只是层级行上的文本字段（隐式创建，见 6.2）。

3.2 BOM 条目

只有两个有效字段：Reference（位号，原样字符串，不解析下划线/后缀）与 Part（器件型号）。

不贴 (DNP) 的表示：某位号「不贴」就是它不出现在该状态的 BOM 中。没有单独贴装标记字段——「无此位号」即「不贴」。

3.3 差量存储与「解析值」

每块单板从初始 BOM（根节点）开始；每个节点 = 一次 commit = 一个 BOM 状态；节点只存相对父节点的修改集（changeset）。单个位号有三种 op：

add 新增 modify 改 Part remove 删除 = 不贴
某位号在某节点的「解析值」（折叠算法的核心定义）：沿父链从根往下，该位号的值 = 离它最近的、显式操作过该位号的祖先节点（含自身）所决定的值：
  • 最近一次显式 op 是 add/modify → 解析为那个 part
  • 最近一次显式 op 是 remove → 解析为「不贴」
  • 一路无任何节点显式操作过 → 继承初始 BOM 值；初始也没有 → 不贴
「显式操作过」= 该节点的 node_changes 里有这条 reference。这是冲突判定的唯一依据。

4 · 数据模型（SQLite）

-- 四级层级，扁平一张表
boards_hierarchy
  id PK · board_name · pcb_version · bom_version · board_uid

-- 初始 BOM，绑定在 BOM版本 层
initial_bom
  id PK · (board_name, pcb_version, bom_version) · reference · part
  UNIQUE(board_name, pcb_version, bom_version, reference)

-- 节点 / commit，含未提交草稿（is_committed=false）
nodes
  id PK (surrogate, 进 URL) · board_id FK · parent_id FK(nullable=根)
  message · created_at · is_committed · committed_at

-- 节点的修改集
node_changes
  id PK · node_id FK · reference · op(add|modify|remove) · part(remove时NULL)
  UNIQUE(node_id, reference)  -- 一个节点对一个位号最多一条

-- append-only 审计日志，永不更新/删除
edit_log
  id PK · node_id · reference · old_part · new_part · op
  source(direct|propagated) · created_at · note
  • 根节点：新建单板ID 时自动以 BOM版本 的 initial_bom 为起点；parent_id=NULL，node_changes 为空（初始 BOM 单独存，不进 changeset）。
  • 工作区：一个 is_committed=false 草稿节点挂在最新已提交节点之后；commit 时翻成正式节点并新开空草稿。
  • UNIQUE(node_id, reference)：保证一个节点对一个位号只解析为一个值——冲突检测的判定依据。

5 · 传播 & 冲突算法 ★（工具核心）

因采用「读取时实时折叠」，传播分两类，处理完全不同。

5.1 编辑历史节点 Sₖ 的某位号 R

步骤动作
① Sₖ 自身写入/更新/删除 Sₖ.node_changes 里 R，记一条 edit_log(source=direct)
② 找下游第一个显式节点 Sⱼ沿子链往后，找 node_changes 里有 R 的第一个 j>k
　└ 没找到自动传播 下游全是「继承」，折叠时自动跟着变，零操作、零冲突
　└ 找到 Sⱼ冲突 Sⱼ 及之后解析为自己的显式值，进入冲突确认
链是线性的，「下游第一个显式节点」最多只有一个，冲突判定干净，不存在多分支合并。

5.2 冲突确认（每个冲突位号二选一）

① 保留下游值

什么都不动。Sⱼ 的显式 op 还在；Sₖ 改 Sₖ，Sⱼ 及之后保持原值。
（那是一次真实的后续改动，与本次修正无关）

② 采用修正值并向后传播

删除 Sⱼ.node_changes 里 R 这一条。删后 Sⱼ 及之后变回「继承」，折叠时自动解析为 Sₖ 新值；给 Sⱼ 记 edit_log(source=propagated)。

三种 op 一视同仁——唯一标准是「下游 changeset 里有没有这条 reference」，与 op 类型无关。编辑根节点改的是 initial_bom 行，逻辑同上；数据访问层统一抽象「取某节点对某位号的显式值」，让根节点与普通节点走同一套代码。

5.3 验收用例（对照需求 4.4）

初始 R1=10k，S2 有 R1=47k，S1/S3 没碰 R1。用户在 S1 写入 R1=22k：

初始
R1=10k
→
S1 ✎
R1=22k
→
S2
R1=47k 显式
→
S3
继承 S2

选「保留下游 47k」

不动 S2。折叠结果：
S1=22k · S2=47k · S3 继承 S2=47k

选「采用修正值」

删除 S2 的 R1。折叠结果：
S1 / S2 / S3 全部继承 = 22k；S2 记 propagated

6 · 页面、路由与交互

服务端渲染 HTML（Jinja2），局部交互用 HTMX 片段替换，几乎不写手写 JS。

6.1 路由表

方法 & 路径用途
GET /层级浏览：四级级联选择
GET /bom-version/new · POST /bom-version新建 BOM版本：上传 CSV（含隐式创建单板名称/PCB版本）
POST /bom-version/import-previewCSV 校验报告（HTMX 局部返回）
POST /board新建单板ID，自动建根节点 + 空工作区草稿
GET /board/{boardId}状态图页面（git graph 风格）
GET /board/{boardId}/node/{nodeId}节点详情/编辑页（稳定 URL）
POST .../node/{nodeId}/edit编辑某位号，触发传播+冲突检测
POST .../node/{nodeId}/resolve提交冲突二选一结果
GET /board/{boardId}/workspace工作区视图（未提交草稿节点）
POST .../workspace/edit在工作区增/改/删位号
POST /board/{boardId}/commit把工作区草稿提交成正式节点
GET /board/{boardId}/log审计日志查询页

6.2 各页面要点

① 层级导航 / 新建 — 新建 BOM版本 在一个表单完成三级：单板名称（下拉或「+新建」）、PCB版本（联动下拉或「+新建」）、BOM版本（输入），上传 CSV → 校验报告 → 确认入库；三元组不重复。无单独的「新建单板名称/PCB版本」入口。新建单板ID 在已存在 BOM版本 下一键创建，自动继承初始 BOM 为根节点。

② CSV 导入（预览 → 确认） — 先 import-preview 解析+校验，预览表展示拆分结果（"R67,R24"|1kR → R67=1kR, R24=1kR），确认后入库。

③ 状态图页面 — 线性节点序列（无分支），每节点显示说明+时间戳，末端工作区草稿虚线区分。纯 HTML/CSS 画竖直 git-graph，不引入图形库。

④ 节点详情 / 编辑页 — 顶部完整 BOM 表（实时折叠）；diff 高亮相对父节点标 新增修改删除；行内编辑 → 跑第 5 节算法 → 无冲突局部刷新 / 有冲突返回确认片段。文案体现「修正记录，非新改动」。

⑤ 冲突确认界面 — 列出所有冲突位号，每行：位号、下游当前值 vs 修正值，二选一单选，一次性提交。

⑥ 工作区 & commit — 复用节点详情渲染（本质是草稿节点），diff 相对最新已提交节点；commit 填说明文字 → 草稿翻正 + 新开空草稿。

⑦ 审计日志页 — 按单板/节点过滤，表格展示节点、位号、旧→新值、op、来源、时间戳、备注。只读。

7 · CSV 导入规则

  • 只取 Reference 和 Part 两列，其余忽略。
  • 合并位号格（如 "R30,R25,R31,R26"）拆成一位号一条，共用同一 Part。
  • 健壮性：UTF-8 BOM 头、CRLF 行尾、含逗号的带引号字段、位号首尾空格。
  • 位号带下划线/后缀（C86_PD1）按字符串原样处理。
  • 校验报告列出：同一位号多次出现、Part 为空、位号为空；用户确认/修正后再入库。

8 · 审计日志（append-only）

  • 每次编辑追加一行，永不覆盖（一个节点改 10 次 = 10 条记录）。
  • 字段：节点、位号、Part 旧值→新值、时间戳、来源（direct / propagated）、可选原因备注。
  • 「来源」解释「为什么这个节点在某次测试后变了」。
  • 暂不做「按时间回看」界面；数据结构保证将来可回放任意时间点状态。

9 · 稳定链接

每个节点有稳定可分享 URL /board/{boardId}/node/{nodeId}，外部笔记粘贴即可打开对应状态完整 BOM。nodeId 是 surrogate key，编辑内容不改 URL。

10 · 测试策略（TDD）

核心逻辑与 Web 层分层测试，重点压在纯逻辑上。

层覆盖重点
① 折叠引擎 ★最重解析值：继承 / add·modify·remove / remove=不贴 / 初始缺失位号
② 传播 & 冲突 ★核心5.3 例子两分支 + 零冲突自动传播 + 多冲突位号 + 编辑根节点 + 三种 op
③ CSV 解析BOM 头 / CRLF / 带引号逗号拆分 / 首尾空格 / 空值与重复进校验报告
④ 审计日志追加不覆盖（改 10 次=10 行）/ direct vs propagated 标记正确
⑤ Web/路由（TestClient）建版本 / 建板自动建根 / commit 翻转 / 稳定 URL / 冲突返回片段
